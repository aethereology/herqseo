## Decisions (append-only ŌĆö the "why" behind the build)

- **D2 ŌĆö Model-agnostic brain via API; never a consumer ChatGPT subscription.** Consumer ChatGPT has no programmatic API and resale violates terms. Default to OpenAI API, route across providers per task. This also mitigates platform-dependency risk.
- **D3 ŌĆö Orchestration layer is the IP.** Multi-tenancy, billing, guardrails, autonomy slider, and integrations are proprietary and get the most engineering care; Hermes is the commodity substrate underneath.
- **D4 ŌĆö Token budgets are first-class.** Per-tier caps enforced in code via a metering wrapper around all model calls. Target 70ŌĆō80% gross margin after API cost.
- **D5 ŌĆö Beachhead ICP is mid-market B2B SaaS (Series AŌĆōC, $5ŌĆō50M ARR, 3ŌĆō15 person marketing teams).** Agencies are the second wedge. Enterprise, sub-$100/mo SMB, and DTC launch are explicitly deferred.

- **D7 ŌĆö Program to an `AgentRuntime` interface, not to Hermes.** Hermes is the first implementation (`HermesAgentRuntime`), not a permanent dependency. Same logic as the model router one layer down: don't couple the product to a young framework's memory model or scheduler before hitting real multi-tenant load. Swapping it later must mean swapping an implementation, not the product.
- **D8 ŌĆö First build is Milestone 0, narrowed to one of everything.** One vertical, one domain, one CMS (WordPress), one autonomy mode (Review), one model path (OpenAI via metering wrapper). M0 proves the loop reliably produces approved, non-breaking, audited, metered work. M0 publishes to staging/draft or our own test sites only ŌĆö **no customer live-site writes until Phase 1**, after the loop is proven. Visibility lift becomes the Phase 1 metric, measured across enough domains/samples to be meaningful.
- **D13 ŌĆö Cross-dialect column types via `with_variant`, not a lowest-common-denominator.** The Python table projections (D12) must run on both SQLite (offline tests) and Postgres (prod). Declaring uuid/enum/timestamptz columns as plain `String` works on SQLite but **fails on Postgres** (no implicit `uuid = varchar`, `text ŌåÆ enum`, or `text ŌåÆ timestamptz` casts) ŌĆö caught only by the Session-20 live-PG validation, and latent even in the pre-existing `SqlBudgetRepository`. Resolution: `sa.String().with_variant(<postgresql type>, "postgresql")` so each column is plain `String` on SQLite (fake ids like `"cp-1"` still round-trip) and the native type on Postgres (so the cast resolves). Helpers in `db.py`: `_UUID`, `_pg_enum(name, *values)` (`create_type=False` ŌĆö the migration owns the type), `_TS`. Lesson: SQLite tests cannot prove Postgres type-compatibility; a CI Postgres service is the real guard.
- **D12 ŌĆö Python DB access is SQLAlchemy Core.** The Python runtime persists via SQLAlchemy Core (sanctioned in tech-stack.md). Columns mirror the `packages/db` migration (SQL migration = source of truth; Python defines minimal table projections, NOT a second ORM schema). Types kept portable so the same code runs on Postgres (prod, with `app.current_org` RLS) and SQLite (offline tests). psycopg3 drives Postgres. First implementation: `SqlBudgetRepository` (the money path).
- **D11 ŌĆö TSŌåöPython boundary is a thin FastAPI service.** The Next.js control plane talks to the Python agent runtime over HTTP via a thin FastAPI adapter (`services/agent-runtime/app.py`) wrapping the framework-free `LoopService`. Rejected: subprocess/CLI bridge (fragile stdout parsing, poor errors) and porting the loop to TS (duplicates logic, violates D1). Keeps Python as the source of truth and containerizable for idle-cheap serverless. Resolves Q2.
- **D10 ŌĆö Use npm workspaces for the initial scaffold.** `pnpm --version` failed in this managed Windows sandbox with `EPERM` while inspecting `C:\Users\kylel`; npm 11 worked and supports the needed workspace scripts. This is a pragmatic local-tooling choice, not a product architecture decision, and can be revisited if pnpm works cleanly outside the sandbox.
